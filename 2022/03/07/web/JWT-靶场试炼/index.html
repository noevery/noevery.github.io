三、JWT - 撤销的令牌

1、题目给出一部分源代码

2、访问给出的靶场地址，能够看出访问login页面方式为POST提交参数，admin页面为GET方式提交参数

3、查看源码，能够看得出是使用flask框架搭建的web页面，查看login页面的代码逻辑，大致意思是没有提交参数username和password参数，就会提示{“msg”: “””Bad request. Submit your login / pass as {“username”:”admin”,”password”:”admin”}”””})，后面代码if username != ‘admin’ or password != ‘admin’，表示如果用户名和密码不等于admin的时候，返回401状态码。说明可以构造{“username”:”admin”,”password”:”admin”}参数提交

4、获得jwt token，源码可以看到生成的jwt token的过期时间为当前时间，时效性为3分钟。说明在生成这个jwt token的时候就已经过期了，既存在黑名单。

5、/admin页面代码逻辑如下：获取jwt token，判断该jwt token是否在/login页面生成的jwt token，不是的话获的flag，也就是说，我们需要一个有效但未发布的 JWT token值。

6、将jwt token算法修改成none，显示不允许指定alg值

7、修改字段exp的时间戳，重新生成jwt token，发包显示签名验证失败，不允许jwt token

8、参考RFC 4648得出，在base64编码下，服务器并不真正关心 BASE64 编码的结果值是 A、A= 还是 A==。也就是说，它们被识别为相同的值。只要在获得jwt token的3分钟内，在jwt token后面添加 = 绕过黑名单，获得flag。
